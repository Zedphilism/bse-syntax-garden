# ACK and NAK

ACK (Acknowledgement) and NAK (Negative Acknowledgement) are control messages used in reliable data transfer: an ACK tells the sender the packet was received correctly; a NAK tells the sender the packet was corrupted and must be retransmitted.

## Explanation

ACK and NAK are the feedback mechanism at the heart of **Automatic Repeat reQuest (ARQ)** protocols. Without them, a sender has no way to know whether its data arrived intact.

**ACK:** "I received the packet correctly — you can send the next one." In cumulative acknowledgement (TCP), the ACK number specifies the next expected byte, not the last received packet number.

**NAK:** "I received something but it was corrupted — please resend." NAKs were introduced in the rdt 2.0 protocol model and eliminated in rdt 2.2, which uses a duplicate ACK for the previous packet as an equivalent signal (if you get a duplicate ACK for packet n−1, that means packet n arrived corrupted).

**Why TCP eliminated NAKs:** having both ACK and NAK types doubles the feedback channel complexity. Duplicate ACKs achieve the same result — and are naturally generated by out-of-order delivery anyway. TCP's fast retransmit (triggered by 3 duplicate ACKs) is a direct consequence of this design.

**ACK in TCP:** TCP uses cumulative ACKs. ACK number N means "I have received all bytes up to N−1; please send byte N next." This is more information-rich than a simple "packet k received" because it covers all bytes, not just the most recent packet.

**In stop-and-wait (rdt 2.x):** a single ACK/NAK is sent per packet. The sender waits, receives either ACK (advance) or NAK (retransmit), and proceeds. In pipelining (GBN, SR), multiple ACKs may be in flight simultaneously.

## Key Points

- ACK: positive confirmation of correct receipt; sender advances to next packet
- NAK: negative acknowledgement; sender retransmits the indicated packet
- rdt 2.0: introduces ACK + NAK; rdt 2.1: adds sequence numbers to handle garbled ACK/NAK
- rdt 2.2 onwards: NAKs eliminated; duplicate ACK for previous packet replaces NAK
- TCP: cumulative ACKs only; no NAKs; duplicate ACKs trigger fast retransmit
- ARQ (Automatic Repeat reQuest): the class of protocols using ACK/NAK + retransmit

## Example

rdt 2.0 flow — packet corrupted:
1. Sender sends PKT with checksum.
2. Receiver detects checksum mismatch → sends NAK.
3. Sender receives NAK → retransmits PKT.
4. Receiver gets correct PKT → sends ACK.
5. Sender advances to next packet.

In TCP (no NAKs):
1. Sender sends segments 1, 2, 3. Segment 2 corrupted.
2. Receiver gets 1 (sends ACK 2), gets corrupted 2 (sends duplicate ACK 1), gets 3 (sends duplicate ACK 1 again).
3. After 3 duplicate ACK 1s → fast retransmit of segment 2.
